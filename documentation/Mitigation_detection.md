## Стратегії протидії (Mitigation & detection)

Для ефективного захисту інформаційної інфраструктури від розроблених нами ж методів прихованої передачі даних необхідно застосовувати концепцію Defense in Depth. Оскільки система використовує легітимні хмарні сервіси, стандартні методи блокування за доменом можуть бути неефективними або заважати робочим процесам.

### 1. Network security

Основним викликом для мережевого захисту є використання протоколу HTTPS та довірених доменів, що зазвичай входять до вайтлістів

* **Впровадження TLS Inspection та DPI:**
    Системи глибокого аналізу трафіку Deep Packet Inspection мають бути налаштовані на дешифрування та перевірку вмісту пакетів (TLS Inspection). Це дозволяє виявляти не просто звернення до `api.telegram.org` або `dropboxapi.com`, а конкретні методи API, що викликаються. Передача зашифрованих бінарних структур усередині JSON-запитів від вузла відправника має стати критичним тригером для системи безпеки.
* **Поведінковий аналіз та моніторинг частоти запитів:**
    Необхідно налаштувати SIEM-системи на виявлення аномальних патернів зв'язку. Регулярні звернення до хмарних сервісів з інтервалом у кілька секунд (навіть із застосуванням випадкових затримок як у нас в останньому варіанті) від вузлів відправників, які не мають відповідного профілю навантаження, повинні генерувати сповіщення. Особлива увага має приділятися робочим станціям, що активують вихідний трафік у неробочий час.

### 2. Host and endpoint Security

Захист безпосередньо на кінцевих точках дозволяє виявити активність скриптів відправника ще до моменту завантаження даних у мережу.

* **Налаштування EDR/XDR систем:**
    Сучасні системи захисту мають бути сконфігуровані на моніторинг активності інтерпретатора Python. Слід впровадити правила, що сигналізують про завантаження специфічних бібліотек (`telethon`, `pyTelegramBotAPI`, `dropbox`) неавторизованими процесами, особливо якщо вони запускаються з тимчасових папок користувача або кореневих директорій дисків.
* **Контроль запуску програмного забезпечення (Whitelisting):**
    Застосування інструментів типу Windows AppLocker для обмеження запуску Python-скриптів. Запуск інтерпретатора має бути дозволений лише для перевірених системних процесів. Для звичайних користувачів можливість виконання `.py` файлів або скомпільованих бінарних файлів, створених через PyInstaller, має бути заблокована за замовчуванням.


### 3. Моніторинг файлової системи 
Алгоритм роботи нашого модуля передбачає специфічну активність у файловій системі відправника перед початком передачі.

* **Аудит операцій з файлами:**
    Налаштування розширеного аудиту (наприклад, Event ID 4663 у середовищі Windows). Системи моніторингу мають реагувати на патерн: послідовне читання конфіденційних даних — створення серії дрібних файлів у директорії `target_folder` — їхнє швидке видалення після отримання підтвердження від отримувача. Така активність є чітким індикатором автоматизованої ексфільтрації.
* **Виявлення нетипових конфігураційних файлів:**
    Автоматизований пошук файлів з розширенням `.env`, що містять чутливі дані (API токени, ідентифікатори Bitwarden). Поява таких файлів у нетипових директоріях є прямим свідченням компрометації вузла відправника.

### 4. Висновки щодо стратегії захисту

Найбільш ефективним методом протидії є комбінація суворого контролю за запуском процесів та детального аналізу мережевих аномалій. Оскільки архітектура системи передбачає динамічне отримання ключів з хмари, відновлення даних після завершення циклу передачі є неможливим, що робить превентивне блокування активності відправника пріоритетним завданням для служб безпеки
